# RIID-4881 / RIID-4917: Codex Auth And Full-Access Harness

[Back to codex-auth-full-access](../codex-auth-full-access.md)

The initial task-scoped `CODEX_HOME` model prevented Codex from reading the
user's global config and removed Codex-managed ChatGPT auth. Real SaaS
assignments then failed with provider 401 responses.

RIID-4881 first kept the real Codex auth store available while trying a
daemon-owned permission profile for provider tool commands. RIID-4917 supersedes
that experiment: daemon no longer owns Codex internal filesystem permission
semantics.

Corrected boundary:

- C6 no longer materializes Codex `.codex/config.toml` or
  `native_config_home=<workdir>/.codex`.
- C4 starts `codex --sandbox danger-full-access app-server --listen stdio://`.
- `danger-full-access` is the only Codex sandbox selection generated by daemon.
- Free-form custom args cannot pass `-c`, `--config`, `--enable`, `--disable`,
  `--sandbox`, `--sandbox=*`, `-s`, `-s=*`, `--yolo`, or
  `--dangerously-bypass-approvals-and-sandbox`.
- Harness owns assignment snapshot, daemon-selected workdir/evidence root,
  provider lifecycle, heartbeat/stale lease handling, terminal result reporting,
  dropped arg evidence, and provider integration gates.
- Codex runtime registration reports host Codex config `model` as the default
  runtime-scoped model catalog entry instead of inventing `codex-default`.

`docs/20-domain/security.md` owns the C7 judgment that full-access/trusted
runtime envelopes are explicit harness-managed choices. `docs/20-domain/provider-runtime.md`
owns provider-by-provider C4 adoption. Codex is currently adopted; Claude,
Cursor, and OpenClaw need their own SSOT, command-builder tests, and real
integration evidence before adoption.

This slice does not move OpenClaw/Cursor adapters, supervisor polling/runtime
selection, SaaS control-plane adapters, task DB/project/mwsd local API packages,
packaging artifacts, private infra, secrets, or local machine state.
